# DNS Blocklists Runbook

This runbook defines how to safely update, validate, release, and monitor
CenterVPN DNS blocklists while preserving the DNS tier promise.

See also:

- `TIER_PROMISE.md`
- `SOURCES_POLICY.md`

## 1) Operating model

- `DNS1`: no blocking baseline.
- `DNS2`: ad + tracker blocking with low-breakage bias.
- `DNS3`: strict mode (DNS2 + malware/social/gambling/privacy telemetry).

Never tune lists in a way that silently changes these product contracts.

## 2) Files you will touch

- `lists/<category>/sources.yaml`: upstream feeds per category.
- `lists/<category>/exemptions.txt`: local allow-exceptions (false positives).
- `configs/d2.toml`, `configs/d3.toml`: tier composition.
- `dist/*`: rendered artifacts (generated by `scripts/render.py`).

## 3) Regular update cadence

- Weekly or biweekly source refresh.
- Monthly quality review (false positives, noisy feeds, large deltas).
- Quarterly tier promise review with product/support stakeholders.

## 4) Standard release checklist

Run from repo root:

```bash
python3 scripts/validate.py
python3 scripts/render.py
```

Then verify:

1. `validate.py` is OK.
2. `render.py` fetches all sources without failures.
3. `dist/manifest.json` updated.
4. `dist/d2.conf` and `dist/d3.conf` show expected delta (not surprising spike).
5. Spot-check sample domains from added/changed sources.

If delta is unexpectedly large:

- pause release,
- inspect changed upstream sources,
- either pin/remove noisy source or move it to stricter tier (usually DNS3).

### Foolproof release command (recommended)

Use the guarded wrapper:

```bash
./scripts/release_blocklists.sh --tag v1.0.2
```

This performs:

- branch guard (`main` only),
- clean-working-tree guard,
- tag format validation,
- duplicate tag checks (local + origin),
- `validate.py` + tagged `render.py --tag`,
- manifest summary output.

When ready to publish:

```bash
./scripts/release_blocklists.sh --tag v1.0.2 --push
```

For non-interactive automation:

```bash
./scripts/release_blocklists.sh --tag v1.0.2 --push --yes
```

## 5) Canary and fleet rollout

1. Release to one canary gateway first.
2. Trigger blocklist sync and verify new manifest hash is active.
3. Check resolver health and admin page status.
4. If healthy, roll out to remaining gateways.
5. Monitor support/error signals for 24 hours.

## 6) False-positive response (fast path)

1. Reproduce blocked domain.
2. Identify likely category causing the block.
3. Add minimal entry to `lists/<category>/exemptions.txt`.
4. Re-run:

```bash
python3 scripts/validate.py
python3 scripts/render.py
```

5. Canary sync first, then fleet rollout.
6. Record reason in commit/PR message.

Guideline: use the narrowest exemption possible (single FQDN over wildcard).

## 7) Rollback procedure

Use one of:

- revert latest commit and re-render, or
- redeploy previous known-good `dist` manifest/hash in your deployment path.

Then trigger sync on canary and verify health before full rollback rollout.

Use as candidate discovery channels (all candidates must pass `SOURCES_POLICY.md`):

- [Mullvad DNS blocklists](https://github.com/mullvad/dns-blocklists)

For malware/phishing enrichment, monitor reputable threat intel feeds and CERT
advisories, then onboard only if licensing, quality, and scope are suitable.

## 9) DNS2 vs DNS3 placement rule

- Prefer new or aggressive feeds in `DNS3` first.
- Promote to `DNS2` only after sustained low-breakage observations.
- If uncertain, default to stricter tier only (DNS3).

## 10) PR template (copy-paste)

Use this in commits/PRs for blocklist updates:

```text
Title: blocklists: refresh <category> sources and render artifacts

What changed:
- Added/updated sources in: <paths>
- Updated exemptions in: <paths or none>
- Re-rendered dist artifacts and manifest

Validation:
- python3 scripts/validate.py (OK)
- python3 scripts/render.py (OK)

Risk:
- Expected impact on DNS2: <low/medium/high + reason>
- Expected impact on DNS3: <low/medium/high + reason>

Rollout:
- Canary gateway first
- Fleet rollout after canary health check
```

## 11) Post-release admin verification

After pushing a new tag:

1. Open admin DNS page.
2. Click `Poll now` (backend picks the new GitHub release immediately).
3. Force sync one canary gateway.
4. Confirm:
   - backend current tag = released tag,
   - canary `loaded_tag` = released tag,
   - recent events include `release_promoted` and `gateway_apply_ok`.
5. Force sync remaining gateways (or wait for timer rollout).
6. Confirm all gateways report `current` lag state.
